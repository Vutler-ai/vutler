# March 2026 release snapshot

This note summarizes the last ~20–30 commits (late March 2026) so the documentation stays aligned with the current code:

## Drive & shared storage

- `api/drive.js` and `services/s3Storage.js` now surface richer metadata (created/modified, MIME types) and support drag/drop move+rename/+delete flows, zero-byte folder markers, and agent writes to index metadata. `frontend/src/app/(app)/drive/page.tsx`, `frontend/src/lib/api/endpoints/drive.ts`, and `frontend/src/lib/api/types.ts` mirror that UX with previews/fullscreen, rename/move+breadcrumb dialogs, badges, and new API helpers.
- Workspace policies now steer drive placement: `services/drivePlacementPolicy.js`, `services/skills/adapters/WorkspaceDriveAdapter.js`, and the workspace provisioning hooks (`app/custom/services/provisioning.js`, `services/workspacePlanService.js`, `api/workspace.js`, `packages/core/middleware/featureGate.js`) synchronize the default root path (`drive_root` in `api/settings.js`) with plan gating so agents, chat, and tasks write to the same shared bucket.

## Session, feature gating, and upgrade flows

- The frontend now stores auth, features, and admin tokens via `/api/session`, `/api/session/features`, and `/api/admin-session`, plus helper files (`frontend/src/lib/auth/session.ts`, `frontend/src/lib/auth/feature-snapshot.ts`). `frontend/src/hooks/useFeatures.ts` writes the snapshot cookie, and `frontend/src/components/layout/app-shell.tsx` uses `frontend/src/lib/auth/route-access.ts` (exposed through `frontend/src/proxy.ts`) to detect guarded prefixes; unauthorized visits now redirect to `/upgrade/<feature>` or `/login` depending on status.
- Server-backed cookies keep feature snapshots and workspace limits in sync with `workspacePlanService.syncWorkspacePlan()`, enabling sidebar/bottom-nav gating (`frontend/src/components/layout/app-sidebar.tsx`, `frontend/src/components/layout/bottom-nav.tsx`), Workspace Plan provisioning (`api/onboarding.js`, `app/custom/services/provisioning.js`), and the new upgrade page (`frontend/src/app/(app)/upgrade/[feature]/page.tsx`) that quotes recommended plans and seeds billing query params.

## Billing, providers & plan data

- `api/billing.js` now normalizes limits (storage, nexus nodes, social posts), ships workspace usage that includes social post quotas, and exposes credit packs via `/billing/credits`. `frontend/src/app/(app)/billing/page.tsx` draws on that data, adds recommended badges from URL params, and replaces the placeholder credit card card with a live `CreditPacks` component plus richer plan limit displays.
- `api/providers.js` has been rewritten to mirror the `tenant_vutler.llm_providers` table, decrypt legacy rows, redact keys safely, and automatically seed Codex/Anthropic/trial providers; `api/auth.js` and the billing webhook insert trial keys via the new table instead of the deprecated `workspace_llm_providers`. The workspace settings API also captures the `drive_root` and `llm_providers` payload used by the UI.

## Nexus, swarm, and agent runtime hygiene

- Nexus observability and billing got new tables (`api/nexus.js`), command health instrumentation, sandbox audit wiring, and Jest isolation (`tests/orchestrator`). `app/custom/services/swarmCoordinator.js`, `app/custom/services/taskExecutor.js`, and `services/taskRouter.js` now resolve workspace placement policies before routing claims so retries, plan quotas, and Swarm moments stay consistent across nodes.
- Agents received standard placement policies (`services/agentConfigPolicy.js` + tests), capability-aware runtime payloads (`dcf7b14`), internal tool quota exemptions, and normalized avatar paths (`lib/avatarPath.js` + `scripts/migrations/20260331_normalize_agent_avatars.sql`) so templates, Workspace-built agents, and Jarvis all fall back to `/sprites/agent-<slug>.png`. `frontend` components (agents list, agent memory, email sidebar, onboarding, dashboard) now use the helper `getAvatarImageUrl`/`isEmojiAvatar` and update `svg`/emoji fallbacks.

## Snipara memory & knowledge

- Snipara’s runtime boundary was refactored into `services/memory/runtime.js`, `modeResolver.js`, `promptBuilder.js`, and the new `services/snipara/gateway.js` so chat/memory agents talk via a single gateway; tests (memory runtime, orchestrator retry, LLM router modes) confirm the new contract. `services/sniparaProvisioningService.js`, `services/sniparaService.js`, and `services/sniparaTaskAdapter.js` automate workspace provisioning, swarm task flow, and Swarm adapters, while new validators (`services/verificationEngine.js`, `services/watchdog.js`) and resolver tweaks (`services/sniparaResolver.js`) keep documents tolerant, ignore transient Snipara errors, and fallback to local SOUL documents (`services/sniparaMemoryService.js`).

## Chat, tasks & workspace overrides

- Workspaces can now override chat tools and task links: `frontend/src/app/(app)/chat/page.tsx` surfaces workspace drives, tasks, and email groups, while `frontend/src/app/(app)/tasks/page.tsx` ships direct artifact links. Workspace placement policies (`services/drivePlacementPolicy.js`) and `services/skills/adapters/*` ensure tasks route to the correct drive adapters and Gmail/Project Management tools; new tests verify artifact placement + workspace defaults.
- Workspace memory views were clarified (`frontend/src/app/(app)/agents/[id]/memory/page.tsx`, `frontend/src/app/(app)/memory/page.tsx`), agent capabilities are injected into chat runtime (`app/custom/services/chatRuntime.js`), and internal tools no longer count against the 8-skill quota (`services/agentConfigPolicy.js`).

## Documentation & cleanup

- The docs index now highlights this snapshot alongside the Nexus Enterprise Virtual Employee brief (`docs/product-briefs/nexus-enterprise-virtual-employee.md`) and the Nexus Enterprise Event Ingestion spec (`docs/specs/nexus-enterprise-event-ingestion-v1.md`). The onboarding workflow (`docs/jarvis-onboarding-workflow.md`) advertises `gpt-5.4-mini` trial tokens, and deprecated mentions of Rocket.Chat, MiniMax, and legacy models have been removed so the docs stay aligned with the current stack described in `AGENTS.md`.
