# Late March 2026 snapshot

Résumé rapide des dix derniers commits pour garder la documentation alignée sur le code (mars 31, 2026).

Update Avril 2026:

- `A0 root orchestration run entrypoints` est maintenant recâblé dans le code courant: les demandes chat multi-étapes, les tâches autonomes reprises par `TaskExecutor`, et les schedules `run_template` passent désormais par un bootstrap commun qui crée ou réutilise un `orchestration_run` durable et projette ses identifiants dans `tasks.metadata`. Résultat attendu: un seul run racine visible par demande autonome au lieu d’un fanout implicite différent selon l’entrée.
- `A1 deferred tool actions -> durable runs` ferme un autre trou d’autonomie: quand `llmRouter` décide qu’un tool call doit sortir du tour synchrone (`async` ou `approval_required`), il peut maintenant créer une vraie tâche racine + un `orchestration_run` durable, garder le `chat_action_run` ouvert en `scheduled` ou `awaiting_approval`, puis laisser le `runEngine` reprendre le cycle `approval -> execute_actions -> finalize`.
- `A2 verification review hardening` ferme un angle mort de fiabilité: les sorties auto-acceptées faute de critères explicites ou parce que le vérifieur LLM était indisponible ne finalisent plus silencieusement un `orchestration_run`. Le moteur dérive d’abord des critères depuis les métadonnées de phase d’orchestration, puis, si la vérification reste non fondée, il ouvre une vraie `approval_gate` avec motif `verification_review` au lieu de considérer le passage comme un succès normal.
- `A3 run-level governance controls` rend enfin les limites d’autonomie exécutoires au runtime: deadlines normalisées, budgets explicites (`max_delegate_tasks`, `max_tool_actions`) et caps de concurrence (`workspace`, `agent`) sont maintenant lus depuis la gouvernance du run, projetés dans `tasks.metadata`, et appliqués par `runEngine`. Effet attendu: un run trop vieux passe en `timed_out`, un run hors budget se bloque explicitement, et un run sans slot disponible se gare en `sleeping` au lieu de continuer en surcapacité.
- `L4 social scheduling visible in Vutler calendar` est maintenant câblé: quand un agent programme une publication sociale différée, `scheduled_at` est propagé en `due_date`, la tâche sociale crée/maintient un événement dans `tenant_vutler.calendar_events`, et les schedules récurrents projettent aussi leur prochain `next_run_at` dans l'agenda Vutler. Résultat attendu: l'agenda devient enfin la source visible de vérité pour les séquences de posts, au lieu de dépendre uniquement des `scheduled_tasks` et de la queue interne.
- `L5 recurring schedule projection + overdue reconciliation` complète le point précédent: l'API calendrier développe maintenant les occurrences futures des `scheduled_tasks` sur la plage affichée au lieu de ne montrer que demain, et le scheduler ajoute une boucle de réconciliation qui exécute les schedules en retard (`next_run_at <= NOW()`) pour éviter de perdre silencieusement un run après un restart ou un raté de timer.
- `L6 finite multi-day requests are materialized upfront` change le contrat d'orchestration: quand l'agent reçoit une demande finie du type "fais ça pendant les 30 prochains jours", le tool de scheduling peut maintenant matérialiser immédiatement les occurrences en vraies tâches datées (`due_date`) avec rappels et événements calendrier associés, au lieu de ne créer qu'un schedule abstrait. Les schedules récurrents restent réservés aux cadences ouvertes du type "tous les jours" sans horizon de fin explicite.
- `L3 drive lanes + social orchestration scheduling` est maintenant cadré dans le code courant: les espaces Drive agents sont réorganisés sous `Agents/<lane>/<agent>` avec script de reclassement, et le runtime d’agent expose enfin le scheduler réel pour les séquences multi-jours. Le correctif ferme aussi deux trous de persistance/exécution: `due_date` et `reminder_at` sont désormais écrits dans les vraies colonnes `tasks`, et `TaskExecutor` ne réclame plus les tâches planifiées dans le futur.
